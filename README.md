# _Disclaimer_
**Цель**: Настройка [`dae`](https://github.com/daeuniverse/dae/tree/main) на `openwrt` для прозрачного проксирования. <br>

**Ограничения**: <br>
На роутере должны быть отключены `hard/soft offloading`, чтобы трафик не уходил мимо dae. <br>
Настройка учитывает только `IPv4`. <br>
geo данные используются из списка [`v2fly`](https://github.com/v2fly/domain-list-community) <br>
Руководство **_не_** включает в себя сборку и установку пакета `dae`. <br>
Учитывайте потребление dae: будет где-то ~130MB ОЗУ в дефолте минусом. Мне, к примеру, хватило отключения uhttpd чтобы был некоторый запас.

Готовый конфиг приложен в репозитарии. Что нужно - правите под себя. <br>
[Общий файл-пример](https://github.com/daeuniverse/dae/blob/main/example.dae) из репозитария проекта dae.

Дефолтный конфигурационный файл должен располагаться как `/etc/dae/config.dae`.

Далее даются лишь некоторые пояснения.

#### 1. Глобальный раздел <br>
`disable_waiting_network: true` <br>
в `default` конфиге `false`, но т.к. дальше в настройках блока `dns` биндим `bind: '127.0.0.1:5353'` на прослушку локальных DNS запросов, чтобы не было deadlock'ов ставим `true`.

`sniffing_timeout: 300ms` у меня сервачёк VPS подальше - ориентируемся на свой пинг до прокси сервера с небольшим запасом.

`so_mark_from_dae: 0` - чтобы исключить разные ситуации с петлями трафика.

`fallback_resolver: '9.9.9.9:53'` запасной DNS сервер, если основной отвалится. Можно опустить настройку, если дефолтный `8.8.8.8:53` устраивает.

#### 2. DNS раздел

`bind: '127.0.0.1:5353'` - лучшее решение, которое я нашёл для себя конкретно указав dae где перехватывать DNS запросы. <br>
Это **_не_** отключает прослушивание DNS запросов помимо этого сокета, т.е. если где-то в трафике есть обращение, например, к `8.8.8.8:53` - dae это тоже перехватит. В большинстве найденных конфигов этот параметр опущен, однако считаю, что лучше явное определение.

В `request` роутинг DNS блока в `qname` с `asis` прописываете свой список сайтов / группы по которым вы ходите через "провайдоровский" или любой другой DNS который у вас используется обычно. 

`fallback: google_DoH` - все запросы не описанные явно - отправляются через этот узел. Если хочется свой DoH, DoT или просто :53 - прописываете свой, предварительно добавив его в [`upstream {`](https://github.com/daeuniverse/dae/blob/030902f519f5b63f839327fd2fa9d8f906f4c504/example.dae#L190). 

Все правила (`qname`, `qtype`) должны идти сверху вниз - от узкого до самого широкого по охвату.

[Полные правила](https://github.com/daeuniverse/dae/blob/main/docs/en/configuration/dns.md) по DNS разделу.

#### 3. Подписки / ноды

Прописываете свою [подписку](https://github.com/daeuniverse/dae/blob/030902f519f5b63f839327fd2fa9d8f906f4c504/example.dae#L145) или [ноду](https://github.com/daeuniverse/dae/blob/030902f519f5b63f839327fd2fa9d8f906f4c504/example.dae#L159) по примерам.

[Список](https://github.com/daeuniverse/dae/blob/main/docs/en/proxy-protocols.md) поддерживаемых протоколов.

Если у вас одна единственная подписка/нода, то в группе указывается

```
group {
    proxy_group { #название группы
        policy: fixed(0) #(номер подписки/ноды, отсчёт от 0)
    }
}
```
Далее в правилах проксирования указывается **_`название группы`_**, а не название саба/ноды! <br>
В целом, настройки групп опускаю, так как у каждого они будут свои в зависимости от подписок/нод. Смотрим в [оф доке](https://github.com/daeuniverse/dae/blob/030902f519f5b63f839327fd2fa9d8f906f4c504/example.dae#L245).

#### 4. Роутинг

Все правила так же должны идти сверху вниз - от узкого до самого широкого по охвату.
При правиле `direct` dae пропускает трафик, но "заглядывает" в него, пополняя свою "БД" соответствием IP <-> domain.

`pname(dnsmasq, dropbear, sshtunnel) -> must_direct` - при `must_direct` dae никак не обрабатывает трафик, сразу выпуская его. <br>
`pname` - это правило для имён процессов в системе где работает dae, т.е. на роутере (не на клиентах!).

`l4proto(udp) && dport(443) -> block` в руководствах рекомендуется блокировать udp на 443 порт из-за его прожорливости по ресурсам CPU. Если у вас с запасом по мощности всё в порядке, наверное, это правило вам не обязательно. Полной картины по этому вопросу у меня нет.

Обязательно пропускаем весь локальный пул без обработки
```
dip(127.0.0.0/8,
    10.0.0.0/8,
    100.64.0.0/10,
    169.254.0.0/16,
    172.16.0.0/12,
    192.0.0.0/24
) -> must_direct
dip('::1/128',
    'fe80::/10',
    'ff00::/8'
) -> must_direct
```

На усмотрение, предпочитаю пускать сами запросы DNS тоже через прокси.

```
domain(geosite:category-doh) -> proxy_group
dip(1.1.1.1, 1.0.0.1) -> proxy_group
dip(8.8.8.8, 8.8.4.4) -> proxy_group
```

`dscp(0x4) -> direct` - сопоставление по [DSCP](https://github.com/daeuniverse/dae/discussions/295), пропуск BitTorrent трафика (и, видимо, не только) напрямую

`fallback: direct` - всё что не описано явно - пускается прямо. Если делаете наоборот `fallback: proxy_group` - не забываете прописать выше в правилах `ip(geoip:ru) -> direct` и убрать/закомментировать правила которые прописаны как `proxy_group`.

[Полные правила](https://github.com/daeuniverse/dae/blob/main/docs/en/configuration/routing.md) роутинга

#### 5. Использование

Из [обсуждения](https://github.com/daeuniverse/dae/issues/79) в [последнем комментарии](https://github.com/daeuniverse/dae/issues/79#issuecomment-1913028442) утверждается, что всё работает "из коробки", но из коробки ничего не заработало. Можно использовать скрипт из первого сообщения в issue, для себя я сделал:
```
uci -q del dhcp.@dnsmasq[0].server
uci add_list dhcp.@dnsmasq[0].server='127.0.0.1#5353'
uci add_list dhcp.@dnsmasq[0].server='1.1.1.1'
uci set dhcp.@dnsmasq[0].noresolv='1'
uci commit dhcp
/etc/init.d/dnsmasq reload
```
Т.е. добавил локальный сокет, который указал в биндинге. 8.8.8.8 добавил только на случай, если почему-то нужно будет остановить dae или он отвалится. noresolv='1' обязательно, чтобы dns запросы не оставались внутри lo, выходили наружу и dae мог их перехватывать.

И после этого использую скрипт из того же последнего комментария - [init скрипт из ImmortalWrt](https://github.com/immortalwrt/packages/blob/master/net/dae/files/dae.init).
Скрипт должен лежать как `/etc/init.d/dae`, не забываем сделать его исполняемым `cmod +x /etc/init.d/dae`.
Скрипт удобно использовать через `hot_reload`, без перезагрузки сервиса dae при внесении каки=то изменений.

При условии, что dae лежит по стандартному `PATH` (`/usr/bin`) без скрипта напрямую dae запускается как: `dae run -c /путь/до/конфига`. Если конфиг лежит как дефолтный `/etc/dae/config.dae` можно опускать `-c` и путь.
Через скрипт: `/etc/init.d/dae start`

#### 6. Допы

Так же здесь в репозитарии небольшой скрипт обновления geo файлов. Кого не устраивают geo файлы от v2fly - замените в скрипте ссылки на свои. Скрипт рекомендуется положить как `/usr/share/dae/geo-update`. Так же не забываем сделать его исполняемым `chmod +x /usr/share/dae/geo-update`.
Прописываем своё правило в планировщик когда будет обновляться: `cron -e`<br>
`0 20 * * 5 /usr/share/dae/geo-update`

#### PS.
При работе локального DHCP ещё и в режиме IPv6 - ничего не работало. После отключения всех связанных IPv6 функций и получения клиентским устройством обычного IPv4 адреса - всё завелось. Имейте ввиду этот момент. Если кто-то знает причину - просьба поделиться в issue.
